# 精准还原 UI 设计工作流

## 任务目标

你要根据用户提供的 UI 设计图还原前端页面。

目标不是做一个风格相似的页面，而是尽量精确还原设计图中的布局、尺寸、位置、颜色、圆角、阴影、透明度、层级、留白和视觉节奏。

## 执行原则

- 不要直接根据图片一次性输出最终代码。
- 必须先把设计图拆成结构化结果，再根据结构化结果生成页面。
- 不要随便进行逻辑兜底。
- 不要在信息不明确时自行补充“更合理”的 UI。
- 不要擅自更换主题色、布局方向、模块顺序、文案层级。
- 不要为了省事把真正需要局部重构的地方继续糊在一个大文件里。

## 前端实现约束

- 优先使用免构建的原生 HTML + CSS + ES Modules。
- HTML 只保留薄壳。
- 页面结构、组件结构、样式、资源引用分别拆分。

## 处理流程

### 第 1 步：读取设计图基础信息

- 记录设计图尺寸。
- 后续所有坐标、尺寸、间距都以设计图原始尺寸为基准。
- 如果设计图外面带有设备边框、状态栏、拍摄背景，要先判断哪些属于页面主体，哪些不属于。

### 第 2 步：先拆一级结构

- 先识别页面骨架，不要一上来细拆按钮和图标。
- 至少输出这些一级结构：
  - 画布
  - 背景层
  - 主内容层
  - 左侧或上方主展示区
  - 右侧或下方主交互区
  - 装饰层
- 如果页面不是左右结构，就按实际视觉骨架拆分。
- 一级结构没有确认前，不要开始写代码。

### 第 3 步：识别真实布局元素和纯视觉元素

必须把元素分成两类：

- 真实布局元素
- 纯视觉元素

真实布局元素指：

- 会参与排版
- 会影响留白和对齐
- 会承载文本或交互
- 后续需要稳定维护

纯视觉元素指：

- 背景图
- 插画
- 光效
- 漂浮装饰
- 纯氛围层

不要把纯视觉元素拆成大量无意义 DOM。
不要把真实布局元素全部写成绝对定位。

### 第 4 步：为每个关键元素记录结构化信息

对每个关键元素记录以下字段：

- `name`
- `role`
- `parent`
- `x`
- `y`
- `width`
- `height`
- `z_index`
- `layout_type`
- `gap`
- `padding`
- `margin`
- `background`
- `border`
- `border_radius`
- `shadow`
- `blur`
- `opacity`
- `font_family`
- `font_size`
- `font_weight`
- `line_height`
- `color`
- `text_align`
- `asset_ref`

坐标原点默认是设计图左上角。

- `x` 向右增大。
- `y` 向下增大。

如果某个值无法精准判断，写“待确认”，不要私自编一个看起来合理的值。

### 第 5 步：先输出结构化规格稿，再进入代码阶段

在生成代码前，先输出一份结构化规格稿。

规格稿至少包含：

- 画布尺寸
- 一级结构划分
- 关键容器边界框
- 文本节点样式
- 主要装饰资源
- 布局方式
- 全局颜色 token
- 全局圆角 token
- 全局阴影 token

结构化规格稿没有完成前，不要跳过直接写页面。

### 第 6 步：决定哪些区域用布局实现，哪些区域用资源实现

下面这类内容优先用正常布局实现：

- 标题
- 副标题
- 表单
- 按钮
- 功能列表
- 卡片容器
- 导航和说明块

下面这类内容优先用资源实现：

- 复杂人物图
- 复杂插画
- 复杂流光
- 高细节光斑
- 难以稳定复刻的氛围背景

不要为了追求“纯代码实现”而牺牲还原度。

### 第 7 步：实现时采用两层结构

第一层是骨架层。

- 用 `flex`、`grid`、常规块布局完成主要结构。
- 主要容器、表单区、标题区、列表区优先使用正常文档流。

第二层是装饰层。

- 只把纯视觉装饰放到绝对定位层。
- 装饰层不要反过来主导骨架布局。

### 第 8 步：样式实现要求

- 所有关键颜色都要从设计图提取，不要自行换色。
- 所有关键圆角都要尽量统一成 token。
- 所有关键阴影都要尽量统一成 token。
- 所有主要间距都要记录数值，不要只写“大概”。
- 透明卡片、磨砂玻璃、发光边缘等效果，要优先还原真实观感，不要随便替换成普通纯色块。

## 不规则边框与 SVG 坐标还原

当任务的核心不是普通页面骨架，而是还原不规则科技边框、HUD 外框、异形卡片描边时，按下面流程执行。

### 适用条件

出现下面任一情况，就不要先猜 `clip-path`：

- 轮廓存在多个切角、缺口、折线、短刻线。
- 重点是边框本体，而不是边框内的文案和交互。
- 用户明确要求“确定坐标”“记录坐标”“用 SVG 绘制边框”。
- 普通圆角矩形或单一 polygon 无法稳定贴近参考图。

### 第 A 步：先确认交付范围

先判断用户要的是哪一种：

- 只还原边框本体
- 边框加白底展示
- 边框嵌入整页 UI

如果用户只要边框本体，不要额外加卡片背景、说明文案、示意性内容。

### 第 B 步：锁定参考尺寸

- 记录原图尺寸。
- `SVG viewBox` 默认与原图尺寸一致。
- 后续所有坐标都以原图左上角为原点。

如果最终渲染在 HTML 页面中，优先让 SVG 自适应缩放，但内部坐标不要改成相对猜测值。

### 第 C 步：先拆轮廓层，不要一口气画成一个对象

至少拆成这些层次：

- `outerFrame`：最外层主轮廓
- `innerFrame`：内侧细轮廓
- `accents`：顶部缺口、侧边折线、底部短线等装饰线
- `chips` / `ticks`：短刻线、小矩形、小斜杠等局部装饰

如果一个边框看起来复杂，优先拆成多个可解释的小轮廓，不要把整个图强行压成一个不可读的大路径字符串。

### 第 D 步：坐标提取采用“机器粗提取 + 人工校正”

允许先用脚本粗提取轮廓，但不能直接把粗提取结果原样交付。

推荐顺序：

1. 用图像脚本获取原图尺寸。
2. 用阈值筛选或颜色范围筛选找到亮边、发光描边的大致区域。
3. 对顶部、底部、左侧、右侧分别做行列扫描，找首个或最后一个命中的高亮像素，得到主边界锚点。
4. 对干扰较大的区域做人工复核，避免把 Logo、按钮、文本发光误当成边框。
5. 把最终点位整理成显式数组，而不是隐藏在一大段生成逻辑里。

如果自动提取受到中间 Logo、文字、按钮高亮干扰，应明确切换成“半自动提取 + 手工定点”，不要继续相信噪声轮廓。

### 第 E 步：实现约束

- 优先使用 `svg` + `polyline` / `path` / `rect`。
- `polyline` 适合主折线路径和短装饰线。
- `rect` 适合短刻线、小芯片、斜放小矩形。
- 发光效果优先用 SVG `filter`，不要只靠粗暴加宽描边。
- 主轮廓和 glow 层分开绘制，方便单独调强弱。

推荐模式：

- 一层 glow 轮廓
- 一层主描边
- 一层内描边
- 若干装饰线层

### 第 F 步：代码组织要求

坐标数据必须可维护。

推荐形式：

- `const outerFrame = [[x, y], ...]`
- `const innerFrame = [[x, y], ...]`
- `const accents = [{ points, kind }, ...]`
- `const chips = [{ x, y, w, h, tilt }, ...]`

不要把所有点直接拼进一个超长 `d=""` 字符串后失去可读性。
不要让最终代码只能靠重新跑脚本才能修改局部点位。

### 第 G 步：边框类任务的校验重点

截图对比时重点检查：

- 外轮廓长宽比例
- 四角切角位置
- 顶部缺口位置和宽度
- 左右折线转折点
- 底部斜切角和短线位置
- 发光强度是否压过主轮廓
- 内外两层描边间距

修正时必须写成可执行偏差，例如：

- 顶边左侧缺口右移 6 像素
- 右侧中段折线过深，回退 4 像素
- 底边右侧短折线偏长 10 像素
- 左下切角起点偏低 8 像素

### 第 H 步：明确禁止事项

- 禁止在异形边框任务里只用一个大致相似的 `clip-path` 就宣称完成。
- 禁止把主轮廓、装饰线、刻线全部糊成一张位图背景。
- 禁止把自动检测出来的噪声点直接当最终坐标。
- 禁止为了省事省略坐标数据，只留下不可读路径。

### 第 I 步：输出要求补充

如果任务核心是异形边框，结构化规格稿里必须额外输出：

- 原图尺寸
- 主轮廓分层结果
- 关键转折点坐标
- 采用 `SVG` 的原因
- 仍待人工确认的局部点位

### 第 9 步：代码组织要求

代码必须可维护。

- 不要把所有内容都塞进一个超大的 HTML 文件。
- 不要把结构、样式、行为混在一个长字符串里。
- 不要通过一大段 `innerHTML` 字符串充当主要架构。

推荐结构：

- `index.html`
- `main.js`
- `pages/`
- `components/`
- `styles/`
- `assets/`

如果页面明显包含多个独立区域，要拆组件。
如果某个区域只是静态片段，可以按约定决定是否拆 fragment。

### 第 10 步：输出第一版页面

第一版页面的目标是：

- 骨架正确
- 主元素齐全
- 位置关系正确
- 样式方向正确

第一版不要为了细节氛围过度堆代码。
先把主结构做准。

### 第 11 步：固定尺寸截图对比

页面写完后，必须按设计图基准尺寸进行截图对比。

对比时重点检查：

- 整体布局比例
- 卡片位置
- 标题位置
- 输入框高度
- 按钮尺寸
- 留白
- 对齐方式
- 圆角
- 阴影
- 透明度
- 装饰层位置

### 第 12 步：根据偏差定点修正

修正时不要笼统说“再优化一下”。

必须按具体偏差修正，例如：

- 右侧卡片偏窄 24 像素
- 标题离顶部偏近
- 登录按钮高度少了 8 像素
- 输入框透明度过低
- 第三方登录区纵向位置偏下
- 装饰音符过大

每一轮修正都应该基于具体偏差，而不是重新猜一版设计。

## 输出要求

当任务开始时，先输出：

- 设计图尺寸
- 页面主体范围判断
- 一级结构拆分结果
- 真实布局元素清单
- 纯视觉元素清单

然后输出：

- 结构化规格稿

然后再输出：

- 实现方案
- 文件结构
- 页面代码

最后输出：

- 截图对比结论
- 当前仍存在的偏差
- 下一轮需要修正的点

## 禁止事项

- 禁止跳过结构化规格稿直接写页面。
- 禁止在缺失信息时脑补设计。
- 禁止把布局问题全部用 absolute 糊过去。
- 禁止把纯视觉资源拆成大量无意义 DOM。
- 禁止为了省事忽略局部重构需求。
- 禁止用“逻辑兜底”“防御性处理”“大概类似”这类模糊说法代替明确判断。
- 禁止输出一个只求差不多、不求可校正的页面。

## 验收顺序

1. 先验收骨架。
2. 再验收主容器。
3. 再验收文本和表单。
4. 再验收圆角、阴影、透明度。
5. 最后验收装饰层和氛围层。

## 如果遇到不确定项

- 明确标记不确定项。
- 给出具体缺失信息。
- 说明它会影响哪一部分还原精度。
- 不要私自改成你认为更合理的设计。
