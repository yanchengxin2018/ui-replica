# 精准还原 UI 设计工作流

## 任务目标

你要根据用户提供的 UI 设计图还原前端页面。

目标不是做一个风格相似的页面，而是尽量精确还原设计图中的布局、尺寸、位置、颜色、圆角、阴影、透明度、层级、留白和视觉节奏。

## 执行原则

- 不要直接根据图片一次性输出最终代码。
- 必须先把设计图拆成结构化结果，再根据结构化结果生成页面。
- 不要随便进行逻辑兜底。
- 不要在信息不明确时自行补充“更合理”的 UI。
- 不要擅自更换主题色、布局方向、模块顺序、文案层级。
- 不要为了省事把真正需要局部重构的地方继续糊在一个大文件里。

## 前端实现约束

- 优先使用免构建的原生 HTML + CSS + ES Modules。
- HTML 只保留薄壳。
- 页面结构、组件结构、样式、资源引用分别拆分。

## 处理流程

### 第 1 步：读取设计图基础信息

- 记录设计图尺寸。
- 后续所有坐标、尺寸、间距都以设计图原始尺寸为基准。
- 如果设计图外面带有设备边框、状态栏、拍摄背景，要先判断哪些属于页面主体，哪些不属于。

### 第 2 步：先拆一级结构

- 先识别页面骨架，不要一上来细拆按钮和图标。
- 至少输出这些一级结构：
  - 画布
  - 背景层
  - 主内容层
  - 左侧或上方主展示区
  - 右侧或下方主交互区
  - 装饰层
- 如果页面不是左右结构，就按实际视觉骨架拆分。
- 一级结构没有确认前，不要开始写代码。

### 第 3 步：识别真实布局元素和纯视觉元素

必须把元素分成两类：

- 真实布局元素
- 纯视觉元素

真实布局元素指：

- 会参与排版
- 会影响留白和对齐
- 会承载文本或交互
- 后续需要稳定维护

纯视觉元素指：

- 背景图
- 插画
- 光效
- 漂浮装饰
- 纯氛围层

不要把纯视觉元素拆成大量无意义 DOM。
不要把真实布局元素全部写成绝对定位。

### 第 4 步：为每个关键元素记录结构化信息

对每个关键元素记录以下字段：

- `name`
- `role`
- `parent`
- `x`
- `y`
- `width`
- `height`
- `z_index`
- `layout_type`
- `gap`
- `padding`
- `margin`
- `background`
- `border`
- `border_radius`
- `shadow`
- `blur`
- `opacity`
- `font_family`
- `font_size`
- `font_weight`
- `line_height`
- `color`
- `text_align`
- `asset_ref`

坐标原点默认是设计图左上角。

- `x` 向右增大。
- `y` 向下增大。

如果某个值无法精准判断，写“待确认”，不要私自编一个看起来合理的值。

### 第 5 步：先输出结构化规格稿，再进入代码阶段

在生成代码前，先输出一份结构化规格稿。

规格稿至少包含：

- 画布尺寸
- 一级结构划分
- 关键容器边界框
- 文本节点样式
- 主要装饰资源
- 布局方式
- 全局颜色 token
- 全局圆角 token
- 全局阴影 token

结构化规格稿没有完成前，不要跳过直接写页面。

### 第 6 步：决定哪些区域用布局实现，哪些区域用资源实现

下面这类内容优先用正常布局实现：

- 标题
- 副标题
- 表单
- 按钮
- 功能列表
- 卡片容器
- 导航和说明块

下面这类内容优先用资源实现：

- 复杂人物图
- 复杂插画
- 复杂流光
- 高细节光斑
- 难以稳定复刻的氛围背景

不要为了追求“纯代码实现”而牺牲还原度。

### 第 7 步：实现时采用两层结构

第一层是骨架层。

- 用 `flex`、`grid`、常规块布局完成主要结构。
- 主要容器、表单区、标题区、列表区优先使用正常文档流。

第二层是装饰层。

- 只把纯视觉装饰放到绝对定位层。
- 装饰层不要反过来主导骨架布局。

### 第 8 步：样式实现要求

- 所有关键颜色都要从设计图提取，不要自行换色。
- 所有关键圆角都要尽量统一成 token。
- 所有关键阴影都要尽量统一成 token。
- 所有主要间距都要记录数值，不要只写“大概”。
- 透明卡片、磨砂玻璃、发光边缘等效果，要优先还原真实观感，不要随便替换成普通纯色块。

### 第 9 步：代码组织要求

代码必须可维护。

- 不要把所有内容都塞进一个超大的 HTML 文件。
- 不要把结构、样式、行为混在一个长字符串里。
- 不要通过一大段 `innerHTML` 字符串充当主要架构。

推荐结构：

- `index.html`
- `main.js`
- `pages/`
- `components/`
- `styles/`
- `assets/`

如果页面明显包含多个独立区域，要拆组件。
如果某个区域只是静态片段，可以按约定决定是否拆 fragment。

### 第 10 步：输出第一版页面

第一版页面的目标是：

- 骨架正确
- 主元素齐全
- 位置关系正确
- 样式方向正确

第一版不要为了细节氛围过度堆代码。
先把主结构做准。

### 第 11 步：固定尺寸截图对比

页面写完后，必须按设计图基准尺寸进行截图对比。

对比时重点检查：

- 整体布局比例
- 卡片位置
- 标题位置
- 输入框高度
- 按钮尺寸
- 留白
- 对齐方式
- 圆角
- 阴影
- 透明度
- 装饰层位置

### 第 12 步：根据偏差定点修正

修正时不要笼统说“再优化一下”。

必须按具体偏差修正，例如：

- 右侧卡片偏窄 24 像素
- 标题离顶部偏近
- 登录按钮高度少了 8 像素
- 输入框透明度过低
- 第三方登录区纵向位置偏下
- 装饰音符过大

每一轮修正都应该基于具体偏差，而不是重新猜一版设计。

## 输出要求

当任务开始时，先输出：

- 设计图尺寸
- 页面主体范围判断
- 一级结构拆分结果
- 真实布局元素清单
- 纯视觉元素清单

然后输出：

- 结构化规格稿

然后再输出：

- 实现方案
- 文件结构
- 页面代码

最后输出：

- 截图对比结论
- 当前仍存在的偏差
- 下一轮需要修正的点

## 禁止事项

- 禁止跳过结构化规格稿直接写页面。
- 禁止在缺失信息时脑补设计。
- 禁止把布局问题全部用 absolute 糊过去。
- 禁止把纯视觉资源拆成大量无意义 DOM。
- 禁止为了省事忽略局部重构需求。
- 禁止用“逻辑兜底”“防御性处理”“大概类似”这类模糊说法代替明确判断。
- 禁止输出一个只求差不多、不求可校正的页面。

## 验收顺序

1. 先验收骨架。
2. 再验收主容器。
3. 再验收文本和表单。
4. 再验收圆角、阴影、透明度。
5. 最后验收装饰层和氛围层。

## 如果遇到不确定项

- 明确标记不确定项。
- 给出具体缺失信息。
- 说明它会影响哪一部分还原精度。
- 不要私自改成你认为更合理的设计。
